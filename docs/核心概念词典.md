# 核心概念词典

> 本词典收录了 Creative-Career-OS 项目中的核心概念，帮助社区成员快速理解项目运作机制和关键术语。

---

## 一、核心协作概念

### 1. 病例 (Case)

**定义**：
病例是 Creative-Career-OS 知识体系的源头，指从业者遇到的真实、具体、已脱敏的困境描述。

**关键特征**：
- ✅ 基于真实经历
- ✅ 已进行匿名化处理
- ✅ 包含具体情境和细节
- ✅ 聚焦于"问题"而非"情绪"

**价值**：
- 是知识生产的原材料
- 帮助他人避坑
- 引发社区讨论和共识形成
- 最终转化为可复用的模组

**示例**：
> "[某新消费品牌]要求先出3个logo方案'看看效果'，未签订合同也未支付预付款。我提供方案后客户失联，两个月后该品牌发布的logo与我的方案高度相似。"

**相关文档**：
- [01-病例提交指南](./01-病例提交指南.md)

### 2. 共识 (Consensus)

**定义**：
共识是社区在公开讨论中，对病例进行分析、诊断后，提炼出的解决问题策略、话术或原则。

**形成过程**：
```
病例提交 → 社区诊断 → 方案提案 → 讨论优化 → 共识确认
```

**关键特征**：
- 经过多轮社区讨论
- 有来自不同背景的验证
- 具体可操作，非空泛建议
- 无重大逻辑漏洞或法律风险

**价值**：
- 连接病例与模组的桥梁
- 集体智慧的结晶
- 可随新案例持续进化
- 为模组生成提供基础

**示例**：
> 针对"版权全部归属甲方"的病例，社区共识可能是："采用分级授权模式，客户获得使用权，设计师保留著作权，双方权益都得到保障。"

**相关文档**：
- [02-共识共建指南](./02-共识共建指南.md)

---

### 3. 模组 (Module)

**定义**：
模组是将共识结构化后形成的可独立使用、版本化管理的"策略单元"，是可以直接拿来使用的"防身工具"。

**模组类型**：
- **合同条款模组**：标准化的合同条款模板
- **报价策略模组**：报价方法和话术
- **沟通话术模组**：应对各种僵局的沟通技巧
- **流程清单模组**：标准化的工作流程

**关键特征**：
- 📋 标准化格式
- 🔄 版本管理
- 📎 可追溯来源
- 🛠️ 即拿即用

**文件命名规范**：
```
[类型]-[功能描述]-v[版本号].md

示例：
- 合同-著作权归属界定-v1.0.md
- 报价-应对无预算试探-v1.0.md
- 沟通-需求模糊项目推进-v1.0.md
```

**模组结构**：
1. 元信息（来源、日期、贡献者）
2. 核心策略
3. 详细内容（多种方案）
4. 使用指南（何时用、何时不用）
5. 话术模板
6. 版本历史
7. 相关资源

**相关文档**：

## 二、安全与隐私概念

### 4. 脱敏 (Anonymization)

**定义**：
脱敏是指在分享病例时，将所有可识别特定个人、公司或项目的信息替换为通用标识的过程。

**必须脱敏的信息**：
| 信息类型 | 错误示例 | 正确示例 |
|:---|:---|:---|
| 公司名称 | [某知名科技公司]、[某电商平台] | [某大型互联网公司]、[某知名电商平台] |
| 人名 | 张经理、李总 | [客户A]、[对接人B] |
| 项目名称 | [某游戏春节活动]、[某电商大促] | [某游戏春节活动]、[某电商大促项目] |
| 具体金额 | 5万元、8000元 | [约五位数预算]、[约中四位数] |
| 具体时间 | 2024年3月15日 | [去年春季]、[约半年前] |
| 地点 | 北京朝阳区、上海陆家嘴 | [一线城市]、[某商业中心] |

**脱敏原则**：
1. **彻底性**：所有可识别信息都要替换
2. **一致性**：同一实体用同一标识
3. **模糊化**：金额、时间用范围代替精确值
4. **去关联性**：防止组合信息推断身份

**价值**：
- 保护隐私，降低法律风险
- 让案例更具普适性
- 鼓励更多人分享真实经历
- 符合法律法规要求

**相关文档**：
- [01-病例提交指南](./01-病例提交指南.md)

---

### 5. 绝对红线 (Absolute Red Lines)

**定义**：
绝对红线是项目不可逾越的三大底线，违反任何一条都可能导致项目无法持续或产生严重后果。

**三大红线**：

#### 红线一：绝对匿名
- 严禁透露任何可识别客户、公司或个人的信息
- 分享前必须完成脱敏处理
- 即使是"大家都知道"的信息也要脱敏

#### 红线二：非法律意见
- 项目所有产出均为"同行经验分享"
- 不构成法律、财务或专业建议
- 重大决策必须咨询持证专业人士

#### 红线三：建设性质则
- 聚焦于"如何解决"，而非情绪宣泄
- 目标是产出可用的模组
- 不建设无解决方案的单纯批判

**违规后果**：
- 内容被要求删除
- 严重者可能被移出社区
- 造成法律后果的需自行承担责任

**相关文档**：
- [法律免责与参考](./法律免责与参考.md)

## 三、项目运作概念

### 6. 飞轮效应 (Flywheel Effect)

**定义**：
飞轮效应是指"病例→共识→模组→新病例"的循环增强机制，每个环节都在推动下一个环节，形成自我强化的知识生产系统。

**飞轮循环**：
```
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│  病例   │ → │  共识   │ → │  模组   │ → │ 新病例  │
│  (Case) │    │(Consensus)│   │(Module) │    │(New Case)│
└─────────┘    └─────────┘    └─────────┘    └─────────┘
      ↑                                        │
      └────────────────────────────────────────┘
              (应用模组避免问题，
               或遇到新问题提交病例)
```

**各环节作用**：
- **病例**：提供原材料，引发讨论
- **共识**：提炼智慧，形成策略
- **模组**：固化产出，便于使用
- **新病例**：应用反馈，持续迭代

**飞轮加速因素**：
- 更多病例提交
- 更活跃的社区讨论
- 更高质量的模组产出
- 更广泛的应用和验证

**价值**：
- 自我强化的知识生产
- 持续进化的策略体系
- 社区活力的源泉

### 7. CCOS-Agent

**定义**：
CCOS-Agent 是 Creative-Career-OS 的AI辅助工具，定位为"辅助脑而非主宰"，帮助用户完成从模糊抱怨到结构化病例的转换，同时严守社区的安全红线。

**核心角色**：
- **新手引导员**：通过问答引导用户描述事件
- **格式助理**：自动生成符合标准的Markdown文档
- **隐私哨兵**：自动识别并替换敏感信息

**工具组件**：
1. **Prompt指南**：指导用户与AI对话，完成病例撰写
2. **模板生成器**：离线网页工具，生成标准化病例模板
3. **敏感词列表**：社区维护的敏感词和替换建议

**使用原则**：
- AI工具是**辅助与可选**的
- 核心协作流程必须发生在人与人之间
- AI生成的内容必须经过人工最终审查
- 工具目标是降低格式负担，而非替代思考

**工作流程**：
```
用户经历 → 打开工具 → 填写信息 → 
工具生成草案 → 用户审查脱敏 → 
复制提交到GitHub Issue
```

### 8. 建设性质则 (Constructive Principle)

**定义**：
建设性质则是社区的核心文化准则，要求所有贡献聚焦于"如何解决"，而非单纯的情绪宣泄或批判。

**核心要求**：

#### DO（应该做）
- ✅ 提供具体的、可操作的建议
- ✅ 分享真实的案例来支撑观点
- ✅ 关注问题的解决方案，而非指责
- ✅ 承认自己的知识边界
- ✅ 帮助新成员理解社区文化

#### DON'T（不应该做）
- ❌ 人身攻击或贬低他人观点
- ❌ 纯粹的情绪宣泄
- ❌ 垄断话语权，排斥不同声音
- ❌ 在讨论中推销个人服务
- ❌ 提供未经证实的法律建议

**建设性表达的框架**：
```
1. 承认问题："我理解这个问题的严重性..."
2. 分析原因："根据我的经验，这可能是因为..."
3. 提供方案："我建议可以尝试..."
4. 分享案例："我之前遇到过类似情况，当时..."
5. 邀请讨论："大家觉得这个方案可行吗？"
```

**价值**：
- 保持社区的专业性和实用性
- 提高讨论质量，产出可用模组
- 营造安全、积极的交流氛围
- 吸引真正想解决问题的成员

**相关文档**：
- [02-共识共建指南](./02-共识共建指南.md)

## 四、社区治理概念

### 9. 角色体系 (Role System)

**定义**：
项目采用基于贡献的自然角色形成机制，而非自上而下的任命制度。

#### 普通参与者
- 阅读病例，参与讨论
- 分享个人经验和见解
- 提出方案建议
- 对其他方案进行补充

#### 活跃贡献者
- 定期参与病例讨论
- 主动整理讨论内容
- 帮助新成员融入社区
- 提出模组优化建议

#### 维护者
- 引导讨论方向，确保建设性
- 整理共识，创建模组草稿
- 审核 Pull Request
- 维护文档结构和质量
- 处理社区争议

**角色晋升路径**：
```
普通参与者 → 活跃贡献者 → 维护者
     │              │            │
     └──────────────┴────────────┘
           (基于贡献自然形成)
```

**成为维护者的标准**：
- 持续参与讨论（至少3个月）
- 提交过被采纳的模组
- 帮助过新成员融入社区
- 对项目有深入理解

### 10. 版本管理 (Version Control)

**定义**：
版本管理是指对模组进行版本编号和历史记录，确保知识的可追溯性和持续进化。

**版本号规范**：
```
v[主版本号].[次版本号]

v1.0：初始版本
v1.1：小幅优化
v2.0：重大更新，可能改变核心策略
```

**版本更新规则**：
- **小修改**：直接提交PR，说明修改理由（v1.0 → v1.1）
- **大修改**：建议新开Issue讨论（v1.x → v2.0）
- **重大更新**：视为新版本，保留旧版本存档

**版本历史记录**：
每个模组文件必须包含版本历史部分：
```markdown
## 版本历史
- v1.0 (2026-02-02)：初始版本，基于病例#001的共识
- v1.1 (2026-03-15)：优化了话术部分，增加了更多场景
```

**价值**：
- 追踪知识演进过程
- 保留历史版本供参考
- 明确最佳实践的变化
- 避免使用过时的策略
