# 共识共建指南

## 什么是"共识"？

**共识**是 Creative-Career-OS 知识加工过程的核心产出。它指的是社区在公开讨论中，对真实故事进行分析、诊断后，提炼出的解决问题策略、话术或原则。共识是连接"真实故事"与"模组"的桥梁。

## 为什么需要共识？

1. **去中心化知识生产**：传统攻略由专家撰写，本项目知识由**所有患者**共同贡献
2. **多元视角**：汇集不同背景从业者的经验和智慧
3. **实践验证**：经过社区讨论和验证的策略更可靠
4. **持续进化**：共识可以随新案例和法规变化而更新

## 共识形成流程

### 第一阶段：真实故事诊断（1-2周）

#### 参与方式
1. **阅读真实故事**：仔细阅读新提交的真实故事，理解背景和困境
2. **提问澄清**：如有不清楚的地方，在评论区提问
3. **分享经历**：如果你有类似经历，分享你的应对方式
4. **分析诊断**：从专业角度分析问题的本质和成因

#### 诊断框架
```
问题本质：这到底是什么类型的问题？
├── 合同陷阱：条款漏洞、权利义务不清
├── 报价问题：价格谈判、预算模糊
├── 沟通僵局：需求变更、进度延误
├── 版权纠纷：知识产权、使用权争议
└── 其他：社保、税务、职业发展等

成因分析：为什么会发生？
├── 信息不对称：客户掌握更多信息
├── 经验不足：从业者缺乏风险意识
├── 行业惯例：某些"潜规则"导致
└── 法律盲区：对法规理解不到位

影响评估：造成了什么后果？
├── 经济损失：直接金钱损失
├── 时间成本：项目延期、重复工作
├── 心理影响：职业倦怠、信任危机
└── 长期影响：对职业发展的影响
```

### 第二阶段：方案提案（2-3周）

#### 提案形式
社区成员可以提出以下形式的解决方案：

1. **策略建议**：具体的应对方法和步骤
2. **话术模板**：可以直接使用的沟通文案
3. **合同条款**：标准化的条款表述
4. **流程优化**：改进工作流程的建议
5. **预防机制**：避免问题发生的前置措施

#### 提案模板
```markdown
## 方案提案：[方案名称]

### 针对真实故事
关联真实故事：#[真实故事编号]

### 核心思路
[一句话概括这个方案的核心逻辑]

### 具体措施
1. [第一步：具体做什么]
2. [第二步：具体做什么]
3. [第三步：具体做什么]

### 话术/条款示例
```
[提供可直接使用的话术或条款文本]
```

### 适用场景
- [场景1]
- [场景2]
- [场景3]

### 风险提示
[使用此方案可能面临的风险或限制]

### 参考依据
- [相关法律条文]
- [行业惯例]
- [成功案例]
```

### 第三阶段：讨论优化（持续进行）

#### 讨论原则
1. **建设性**：聚焦于改进方案，而非批评个人
2. **具体性**：提供具体的修改建议，而非泛泛而谈
3. **证据性**：有案例或法规支撑的观点更有说服力
4. **包容性**：尊重不同背景和经验带来的多元视角

#### 讨论技巧
- **补充**："这个方案很好，我建议再补充一点..."
- **修正**："关于XX部分，根据我的经验，实际情况可能是..."
- **提问**："如果遇到XX情况，这个方案是否适用？"
- **验证**："我用过类似的方法，效果是这样的..."

### 第四阶段：共识确认（里程碑）

#### 共识标准
一个方案要成为"共识"，需要满足：
1. **多轮讨论**：经过至少2-3轮社区讨论
2. **多元验证**：有来自不同背景的成员验证
3. **无重大异议**：没有明显的逻辑漏洞或法律风险
4. **可执行性**：方案具体、可操作，而非空泛建议

#### 共识标记
当讨论达到共识时，维护者会：
1. 在评论区添加 `共识已形成` 标签
2. 整理共识内容到置顶评论
3. 创建对应的模组文件草稿
4. 关闭该真实故事的讨论（或转为模组优化讨论）

## 模组生成流程

### 从共识到模组

当共识形成后，需要将其转化为标准化的"模组"：

#### 模组文件结构
```markdown
# [模组名称] v[版本号]

## 元信息
- **来源真实故事**：#[真实故事编号]
- **共识形成日期**：[YYYY-MM-DD]
- **最后更新**：[YYYY-MM-DD]
- **贡献者**：[贡献者名单]
- **适用场景**：[场景描述]

## 核心策略
[2-3句话概括核心思路]

## 详细内容
### 方案A：[方案名称]
[详细描述]

### 方案B：[方案名称]
[详细描述]

## 使用指南
### 何时使用
- [适用情况1]
- [适用情况2]

### 何时不适用
- [不适用情况1]
- [不适用情况2]

### 注意事项
- [风险提示1]
- [风险提示2]

## 话术/条款模板
```
[可直接复制使用的模板]
```

## 版本历史
- v1.0 ([日期])：初始版本，基于真实故事#[编号]的共识
- v1.1 ([日期])：优化了XX部分，增加了XX内容

## 相关资源
- [关联真实故事]
- [相关法律条文]
- [延伸阅读]
```

#### 提交模组
1. 在本地创建模组文件
2. 提交 Pull Request 到 `toolkits/` 目录
3. 在 PR 描述中关联原始真实故事和讨论 Issue
4. 等待社区审核和合并

## 社区讨论规范

### DO（应该做）
- ✅ 尊重每一位参与者的经验和观点
- ✅ 提供具体的、可操作的建议
- ✅ 分享真实的案例来支撑观点
- ✅ 关注问题的解决方案，而非指责
- ✅ 承认自己的知识边界，不确定时明确说明
- ✅ 帮助新成员理解讨论上下文

### DON'T（不应该做）
- ❌ 人身攻击或贬低他人观点
- ❌ 提供未经证实的法律或商业建议
- ❌ 泄露任何可能识别个人或公司的信息
- ❌ 纯粹的情绪宣泄，不提供建设性意见
- ❌ 垄断话语权，排斥不同声音
- ❌ 在讨论中推销个人服务或产品

## 角色与职责

### 普通参与者
- 阅读真实故事，参与讨论
- 分享个人经验和见解
- 提出方案建议
- 对其他方案进行补充和修正

### 活跃贡献者
- 定期参与真实故事讨论
- 主动整理讨论内容
- 帮助新成员融入社区
- 提出模组优化建议

### 维护者
- 引导讨论方向，确保建设性
- 整理共识，创建模组草稿
- 审核 Pull Request
- 维护文档结构和质量
- 处理社区争议

## 质量保障机制

### 内容审核
1. **自动检查**：敏感词过滤、格式检查
2. **社区审核**：成员相互监督，发现问题及时指出
3. **维护者审核**：关键决策由维护者把关

### 版本管理
- 每个模组都有版本号
- 重大更新需要重新经过共识流程
- 过时模组移入存档区，但保留参考价値

### 反馈循环
- 使用者可以在 Issue 中反馈模组效果
- 定期回顾和优化热门模组
- 根据新案例更新模组内容

## 常见问题 FAQ

**Q: 我的建议没人回应怎么办？**
A: 优质内容需要时间来被发现。你可以：
- 在合适的时机（如周末）重新分享
- 用更清晰的结构呈现你的建议
- 主动邀请相关领域的成员参与讨论

**Q: 和别人观点不一致怎么办？**
A: 这是正常的，也是社区价值所在。建议：
- 先理解对方观点的出发点和依据
- 提供你的案例或证据支撑你的观点
- 寻求共识，或提出"多方案并存"的建议
- 保持尊重，避免争论升级

**Q: 如何成为维护者？**
A: 维护者由长期、高质量贡献者自然形成。标准包括：
- 持续参与讨论（至少3个月）
- 提交过被采纳的模组
- 帮助过新成员融入社区
- 对项目有深入理解

**Q: 共识形成后还能修改吗？**
A: 可以。模组是持续进化的：
- 小修改：直接提交 PR，说明修改理由
- 大修改：建议新开 Issue 讨论
- 重大更新：视为新版本，保留旧版本存档
