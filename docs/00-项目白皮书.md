# Creative-Career-OS 项目白皮书

## 一、项目缘起与核心哲学

### 1.1 问题根源
创意行业（尤指设计、新媒体艺术、自由职业）存在一个核心矛盾：个体从业者的专业技能（"硬技能"）与行业生存智慧（"暗知识"）严重失衡。后者包括合同风险识别、价格谈判、版权捍卫、客户沟通等，通常通过私人关系或痛苦试错获得，形成了不透明的"信息壁垒"，导致新人成为系统性风险的主要承担者。

### 1.2 从批判到建设
项目始于对行业资源分配不公的批判，但迅速转向更具建设性的目标：**不为批判而批判，而为解法而共建**。我们认为，最高效的抵抗不是呐喊，而是为同行锻造一套趁手的"防身工具"。

### 1.3 核心隐喻：智力基础设施
我们将项目定位为行业共用的"智力基础设施"。如同物理世界的基础设施降低交易成本，本项目旨在通过开源、协作的方式，降低行业内的"信任成本"与"试错成本"，将分散、隐性的生存知识，转化为结构化、可检索、可迭代的公共资产。

## 二、核心协作系统与文档架构

### 2.1 核心协作飞轮
1. **病例**：一切知识的源头。指从业者遇到的真实、具体、已脱敏的困境描述（如合同陷阱、报价难题）。
2. **共识**：加工过程。社区在公开讨论中，对病例进行分析、诊断，提炼出解决问题的策略、话术或原则。
3. **模组**：固化产出。将共识结构化，形成可独立使用、版本化管理的"策略单元"，存入公共工具箱。

### 2.2 三层文档架构
```
Creative-Career-OS/
├── 📄 README.md                          # 门户：快速入门与价值宣言
├── 📁 docs/                              # 手册：核心概念与深度指南
│   ├── 00-项目白皮书.md                  # 哲学、设计与治理
│   ├── 01-病例提交指南.md                # 如何贡献一份合格的"病例"
│   ├── 02-共识共建指南.md                # 如何参与讨论与提炼模组
│   └── 99-外部资源索引.md                # 法律、报告、博主等延伸学习资料
├── 📁 toolkits/                          # 武器库：即拿即用的策略模组
│   ├── 报价-应对无预算试探-v1.0.md
│   ├── 合同-著作权归属界定-v1.1.md
│   └── 沟通-需求模糊项目推进-v1.0.md
└── 📁 tools/                             # 工坊：辅助工具
    └── ccos-agent/                       # AI辅助工具包
        ├── prompt-guide-zh.md            # 大模型Prompt指南
        └── template-generator.html       # 离线病例模板生成器
```

### 2.3 绝对红线：隐私与安全
- **全流程脱敏**：从病例提交的第一刻起，所有可识别信息（公司名、人名、项目名、金额）必须被替换为通用标识（如`[某大型游戏公司]`、`[客户A]`、`[预算约五位数]`）。
- **非法律意见**：项目所有产出均为**同行经验分享**，不构成法律、财务或专业建议。重大决策必须咨询持证专业人士。
- **建设性质则**：聚焦于"如何解决"，禁止人身攻击与单纯的情绪宣泄。不建设无解决方案的批判。

## 三、十轮深度思考与设计加固

项目设计经历了十轮自我批判，以确保其坚实性、安全性与可持续性。

| 思考轮次 | 核心挑战与批判 | 设计回应与加固措施 |
|:---|:---|:---|
| **第一轮** | **用户茫然**：项目概念抽象，不知如何入手。 | 在README顶部设计"五分钟快速启动"，将最简路径（提交/阅读病例）前置。口号直击痛点："别再孤身试错"。 |
| **第二轮** | **信任缺失**：凭什么相信社区内容？缺乏权威性。 | 建立透明溯源机制：每个"模组"必须链接至产生它的"病例"及公开讨论过程，知识来源可追溯。 |
| **第三轮** | **贡献门槛高**：术语（如Issue、PR）和技术流程吓退非技术用户。 | 提供纯手动模板与**可选**的AI辅助工具（`ccos-agent`），将贡献路径分级。撰写平民化指南，解释核心协作概念。 |
| **第四轮** | **隐私风险**：分享案例可能导致个人信息或客户信息泄露。 | **确立"绝对脱敏"为第一铁律**。在《病例提交指南》中提供详尽的脱敏范例和检查清单。工具设计优先考虑离线处理。 |
| **第五轮** | **内容质量滑坡**：低质量或虚假信息污染知识库。 | 设计"病例->共识讨论->模组草案->社区评议->正式入库"的流程，让质量在公开、多轮的互动中涌现，而非依赖单一审核。 |
| **第六轮** | **社区治理困境**：出现争议时，谁来决定？如何防止小团体垄断话语？ | 采用"基于活跃贡献的温和引导"模式。维护者角色由长期、高质量贡献者自然形成，核心职责是整理与提议，而非裁决。所有决策在公开场域进行。 |
| **第七轮** | **知识过时与僵化**：模组一旦写入，可能不再适用，或成为新教条。 | **引入版本管理**。每个模组文件必须包含"版本历史"。鼓励社区根据新案例提出修订。设立"存档"区存放过时但供参考的模组。 |
| **第八轮** | **可持续性**：如何激励长期贡献？项目如何冷启动？ | 设计"首个共建任务"机制，由发起者提供高质量种子病例。建立贡献者荣誉墙（README致谢）。核心激励在于**解决自身问题并见证其成为公共资产**的独特成就感。 |
| **第九轮** | **技术依赖与异化**：过度依赖AI或复杂工具，背离以人为本的初心。 | 明确AI工具（`ccos-agent`）的**辅助与可选**定位。核心协作流程（讨论、共识）必须发生在人与人之间。工具的目标是降低格式负担，而非替代思考。 |
| **第十轮** | **哲学自洽**：此项目是否在构建另一个"系统"，是否违背了反异化的初衷？ | 明确本系统是"**对抗性基础设施**"。它的权力是分布式、透明的，其规则（如何贡献、如何脱敏）由社区共同维护且可修改。其目标是赋能个体，而非规训个体。 |

## 四、启动路线图与初始工具包

### 4.1 初始准备 (Week 0)
1. 在GitHub创建组织 `Creative-Career-OS` 及主仓库。
2. 按第二部分架构，创建目录和核心框架文件。
3. 在 `docs/01-病例提交指南.md` 中，填充详尽的**脱敏范例**（例：将"为[某知名手机品牌]的H5项目设计主视觉，预算8万"作为标准范例）。
4. 在 `toolkits/` 中，创建3-5个"种子模组"（可基于公开资料或发起者经验编写），展示项目潜力。

### 4.2 社区冷启动 (Week 1-4)
1. 发布 **Issue #1: [首个共建任务] 我们如何应对"先做一稿看看，合适再付费"？**。
2. 由发起者及核心贡献者，按照指南提交2-3份高质量的"种子病例"，并积极参与讨论。
3. 将初步共识，整理成第一个共建的"模组"草稿，提交Pull Request，完成首次飞轮闭环。
4. 将项目分享至目标社群，邀请更多"玩家"入局。

### 4.3 首版AI辅助工具 (`ccos-agent v0.1`)
- **定位**：一个引导用户完成病例撰写的"填空题"助手。
- **核心**：
    1. **`prompt-guide-zh.md`**：一份能与任何大语言模型对话的指导手册，通过预设问题引导用户一步步回忆和描述案例，并自动提示脱敏要点。
    2. **`template-generator.html`**：一个离线单文件网页，用户填写关键信息后，自动生成格式规范的病例Markdown草案。
- **原则**：该工具生成的文本必须经用户**人工最终审查与脱敏确认**后方可提交。

## 五、总结与展望

**Creative-Career-OS** 不仅仅是一个开源文档项目。它是一次社会实验，旨在测试：
> 在缺乏传统权威与中心化机构的领域，一群原子化的个体，能否通过理性的规则设计、严谨的隐私保护与开源协作技术，共同构建一个旨在保护彼此、赋能彼此的可持续知识共同体？

它的成功不在于规模，而在于**有效性**——当一位新手凭借从这里获得的一个"模组"，有底气地在谈判中守住底线时；当一位资深从业者的一次惨痛教训，被转化为让百人避坑的公共策略时，这个系统便实现了其最根本的价值。

**项目已准备就绪，等待它的第一批"建造者"。**
