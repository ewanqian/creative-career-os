# 应对项目范围蔓延话术模组 v1.0

## 元信息
- **来源真实故事**：#005（需求从logo设计扩展到全套VI的谈判实录）
- **共识形成日期**：2026-02-02
- **最后更新**：2026-02-02
- **贡献者**：Creative-Career-OS 社区
- **适用场景**：项目进行中客户不断提出新增需求

## 核心策略
通过**范围冻结机制**、**变更管理流程**、**额外收费话术**，防止"项目越做越大，预算一分不加"的困境，确保劳动价值得到合理回报。

## 详细内容

### 方案A：预防式范围界定

**核心思路**：在项目启动前就与客户明确界定项目范围，并建立范围变更的管理机制。

**合同条款模板**：
```
第X条 项目范围与变更管理

1. 项目范围
   本项目包含以下内容（详见附件《项目需求说明书》）：
   a) [具体交付物1]
   b) [具体交付物2]
   c) [具体交付物3]
   
   以下内容不在本项目范围内：
   a) [明确排除的内容1]
   b) [明确排除的内容2]

2. 范围冻结
   a) 合同签订后，项目范围视为冻结
   b) 任何超出约定范围的请求视为"变更请求"
   c) 变更请求需经双方书面确认后方可执行

3. 变更管理流程
   a) 客户提出变更请求（书面形式）
   b) 乙方评估变更影响（工作量、工期、费用）
   c) 双方协商变更方案
   d) 签署变更确认书
   e) 支付变更费用（如适用）
   f) 执行变更

4. 变更费用计算
   a) 小型变更（工作量<4小时）：累计不超过[X]小时免费，超出部分按[X]元/小时计费
   b) 中型变更（工作量4-16小时）：按[X]元/小时计费，或协商固定费用
   c) 大型变更（工作量>16小时）：视为新增项目，重新报价

5. 工期调整
   因变更导致的工期延长，乙方不承担责任，交付时间相应顺延。
```

### 方案B：过程中范围控制

**核心思路**：当客户在项目进行中提出新增需求时，使用专业话术进行管理和引导。

**话术模板**：

**场景1：客户提出"小需求"**
```
收到您的需求。让我先评估一下这个"小调整"的工作量：

[具体需求]涉及：
1. [工作项1]：约[X]小时
2. [工作项2]：约[X]小时
3. [工作项3]：约[X]小时

总计约[X]小时的工作量。

**处理方案：**
- 方案A：纳入本次项目，费用增加[X]元，工期延长[X]天
- 方案B：作为二期项目，本次先按原范围完成，二期单独规划
- 方案C：简化需求，只保留核心部分，费用增加[X]元

您倾向于哪种方案？
```

**场景2：客户说"就加个小东西，很快的"**
```
理解您的需求。不过根据经验，这个"小东西"涉及：

1. [具体工作内容1]
2. [具体工作内容2]
3. [具体工作内容3]

虽然看起来简单，但为了保证质量，需要[X]小时的工作量。

**我的建议：**
- 如果确实需要，可以按[X]元/小时计费
- 或者我们可以先完成原定的核心内容，确保按时交付
- 这个新增需求可以作为优化项，在二期中重点处理

您觉得哪种方式更合适？
```

**场景3：客户不断提出新想法**
```
感谢您对这些细节的关注，说明您很重视这个项目。

我注意到我们已经讨论了多个新增方向。为了确保项目顺利推进，我建议：

**Step 1：范围确认**
让我们先锁定当前已确定的内容，确保这部分按时高质量交付。

**Step 2：变更整理**
把您想到的所有优化点整理成清单。

**Step 3：优先级排序**
一起评估哪些是必须的，哪些是锦上添花的。

**Step 4：二期规划**
把非核心的优化点放到二期，单独规划时间和预算。

这样既保证本期按时交付，又不遗漏您的想法。您看可以吗？
```

### 方案C：紧急止损话术

**核心思路**：当范围蔓延已经严重影响项目时，使用果断但专业的话术进行止损。

**话术模板**：
```
[客户名称]您好，

我想和您沟通一下项目现状。

**当前情况：**
- 原定项目范围：[原范围描述]
- 已确认的变更：[变更清单]
- 当前累计增加工作量：[X]小时
- 对工期影响：延期[X]天

**风险评估：**
如果继续接受新增需求，可能导致：
1. 项目无法按时交付
2. 质量难以保证
3. 双方成本都超支

**我的建议：**
1. 立即冻结范围，专注完成已确定内容
2. 已超出的[X]小时，按[X]元/小时结算
3. 其他想法放到二期项目规划

这样既对本期项目负责，也为后续合作留出空间。

期待您的理解和支持。
```

## 使用指南

### 何时使用
- ✅ 项目进行中客户提出新增需求
- ✅ 客户认为"小改动"不应该加钱
- ✅ 项目范围不断扩大，工期严重延误
- ✅ 需要重新建立项目边界和规则

### 何时不适用
- ❌ 变更确实是由于乙方设计缺陷导致
- ❌ 变更是双方共同认可的合理优化
- ❌ 长期合作中的微调（应在年度合同中约定）

### 注意事项
1. **及时记录**：所有变更请求都要书面记录
2. **量化影响**：用具体的工作量、工期、费用说话
3. **提供选择**：不要只说"不行"，要给出可行方案
4. **保持专业**：避免指责客户，聚焦于解决问题
5. **文档留痕**：所有范围变更都要有书面确认

## 识别范围蔓延信号

以下信号表明项目可能出现范围蔓延：
- ⚠️ 客户频繁提出"小想法"和"小调整"
- ⚠️ "就改一下"变成了系统性的修改
- ⚠️ 项目工期已经延长，但交付物还没确定
- ⚠️ 客户说"既然都做了，不如把XX也做了"
- ⚠️ 乙方已经开始免费加班来满足新增需求
