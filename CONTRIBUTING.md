## 📝 分享经验/案例

你的实战经验对其他从业者很有价值！

### 内容标准

✅ **适合分享的内容**：
- 具体的客户开发案例（脱敏后）
- 定价谈判的实战经验
- 合同陷阱的教训
- 项目管理的技巧
- 职业转型的经历
- 工具/工作流分享

❌ **不适合的内容**：
- 空洞的理论
- 未经实践的方法
- 过于个人化的故事
- 广告或推广

### 格式模板

使用以下模板撰写你的经验：

```markdown
# [标题]

**分享者**：[你的名字/昵称]
**职业**：[你的职业]
**经验年限**：[X年]
**适用阶段**：[生存期/成长期/影响力期]

---

## 背景

**当时的情况**：
[描述背景]

**面临的挑战**：
[描述挑战]

## 解决方案

**我做了什么**：
1. [步骤1]
2. [步骤2]
3. [步骤3]

**关键决策**：
[描述关键决策点]

## 结果

**最终成果**：
[描述结果]

**学到的教训**：
- [教训1]
- [教训2]

## 给其他从业者的建议

1. [建议1]
2. [建议2]
3. [建议3]

---

*经验分享 —— CCOS 创意职业操作系统*
```

### 提交流程

1. **Fork 仓库**
   ```bash
   # 点击 GitHub 页面的 "Fork" 按钮
   ```

2. **创建分支**
   ```bash
   git checkout -b case/你的经验标题
   ```

3. **添加文件**
   - 将文件放在合适的目录
   - 命名规范：`case-简短描述.md`

4. **提交更改**
   ```bash
   git add .
   git commit -m "案例: 添加[标题]经验分享"
   git push origin case/你的经验标题
   ```

5. **创建 Pull Request**
   - 访问你的 Fork 仓库
   - 点击 "Compare & pull request"
   - 填写 PR 描述
   - 提交 PR

## 🛠️ 提供工具/模板

你有好用的工具或模板？分享给大家！

### 工具标准

✅ **适合的工具**：
- 解决具体问题
- 可立即使用
- 有使用说明
- 经过实践验证
- 通用性强（不只适合你个人）

❌ **不适合的工具**：
- 过于个性化的设置
- 需要特定软件/环境
- 没有说明文档
- 未经测试

### 工具类型

| 类型 | 示例 | 存放位置 |
|:---|:---|:---|
| **文档模板** | 报价单、合同、邮件 | `tools/模板/` |
| **表格工具** | 项目跟踪表、财务表 | `tools/表格/` |
| **检查清单** | 合同审查、项目验收 | `tools/清单/` |
| **提示词** | AI辅助提示词 | `docs/AI提示词/` |
| **脚本/代码** | 自动化脚本 | `tools/脚本/` |

### 格式要求

每个工具必须包含：

1. **工具文件**（如 `报价单模板.md`）
2. **使用说明**（在同一文件或单独文件）
3. **示例**（展示如何使用）

### 提交步骤

同"分享经验"的流程，但分支命名为：`tool/工具名称`

## 📚 完善文档

发现文档有错误或可以改进？欢迎修改！

### 可以做的改进

- 修复错别字或语法错误
- 补充缺失的信息
- 添加更多示例
- 改进表述清晰度
- 更新过时的内容
- 添加图片/图表

### 修改流程

1. **找到要修改的文件**
2. **Fork 仓库**
3. **创建分支**：`git checkout -b docs/修改描述`
4. **修改文件**
5. **提交 PR**

### 文档规范

**格式要求**：
- 使用 Markdown 格式
- 标题使用 `#` 层级
- 代码块标注语言
- 链接使用相对路径

**内容要求**：
- 简洁明了
- 具体可操作
- 有实际案例
- 考虑不同经验水平的读者

## 🔧 修复问题

有技术能力？帮忙修复 Bug！

### 开始之前

1. **查看 Issues**：找到标记为 "good first issue" 的问题
2. **评论说明**：在 Issue 下评论说明你要修复
3. **等待确认**：等待维护者确认后再开始

### 修复流程

1. **Fork 仓库**
2. **创建分支**：`git checkout -b fix/issue编号`
3. **修复问题**
4. **测试验证**
5. **提交 PR**

### PR 要求

- 描述清楚修复了什么
- 关联相关的 Issue
- 如果是重大修改，先讨论再实施
